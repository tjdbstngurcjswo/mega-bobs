# 식전 픽 투표 (2단계 투표) — 디자인 스펙

> 상위 문서: Notion 「기획 문서: megabobs.com Phase 1」 F2(메뉴 투표) 확장
> 관련: docs/design/2026-06-05-phase1-design.md (디자인 시스템), docs/design/mockups/home-b.html

## 목적

기존 투표(식후 👍/👎)는 누르고 끝 — 사용자에게 돌아오는 가치가 없다.
투표를 **식전 픽 + 식후 평가** 2단계로 분리해:
1. **점심 의사결정 도움**: 식전 픽의 실시간 분포 = "동료들은 뭘 고르나" 신호
2. **참여 동기/재미**: 픽 vs 만족도 갭 공개 ("어제의 정답은 A")
3. ~~전략 보너스: 나가서 픽 → 맛집 링크~~ (2026-06-05 제거 — '나가서' 옵션은 수요 신호 데이터로만 활용, UI 직접 링크 없음)

## 사용자 흐름

**시간 정책 (2026-06-05 확정): 경계 = 구내식당 마감 13:00. 픽과 맛평가는 절대 공존하지 않는다.**

| 시간대 | 픽 ("오늘 점심은?") | 맛평가 (👍/👎) |
|---|---|---|
| 00:00 ~ 13:00 (마감 전) | 노출 + 선택·변경 가능, 실시간 분포 | **비노출** (아직 먹기 전) |
| 13:00 ~ 24:00 (마감 후) | **UI 제거** (행 자체 사라짐) | 노출 + 선택 가능 + 실시간 결과 |
| 자정 이후 (다음날·과거 조회) | — | 선택 불가, **결과만 읽기 전용** ("맛평가 결과 89% · 27명") |

```
[00:00 ~ 13:00]
보드 하단 픽 행 — 2-상태:
  • 투표 전: "오늘 점심은?" + 버튼 4개만. 분포 비공개 (밴드왜건 방지 + 참여 유도)
  • 투표 후: 버튼 제거 → 결과 스택 바만 ("✓ 코스 A 64% | 18% | 6% | 12%" — 내 픽 세그먼트 옐로우+체크, 나머지 그레이 단계) + "N명 참여"
  → 마감 전 픽 변경: 결과 바의 내 세그먼트 탭 → 버튼 상태로 복귀 (재선택)
  → 코스 행에는 맛평가 UI 없음 (순수 메뉴판)

[13:00 ~ 24:00]
픽 행 제거. 코스 행 좌하단에 맛평가 버튼 + 실시간 결과 노출

[자정 이후]
맛평가 버튼 제거 — 결과 텍스트만. (다음날 갭 카드: "어제 A 픽 64% → 만족 89%")
자동 블로그 글에 픽률 vs 만족률 비교 섹션
```

- 중복 방지: 픽·평가 모두 localStorage UUID 기준 1일 1회 (soft-limit, 기존 F2와 동일)
- 시간 경계: 1차는 클라이언트 시각(Asia/Seoul) 기준 노출 제어 + **RPC에서 서버 시각(now())으로 윈도우 재검증** — 시계 조작/탭 방치로 마감 후 픽이 들어오는 것 차단
- 마감 시각은 운영시간 데이터(13:00)에서 파생 — 하드코딩 금지

## UI (디자인 시스템 준수 — 옐로우 키컬러/풀 스퀘어/고딕)

- **픽 위젯 (한 줄 컴팩트, ~13:00 한정, 2-상태)**: 메뉴보드 하단 1행 — 투표 전 버튼만 / 투표 후 결과 스택 바만(맛집 링크 없음). 마감 후엔 행 자체 제거 — '오늘 점심은?' 라벨 + 소형 버튼 4(버튼 안 % 없음) + 우측 결과 텍스트('A 64% · 나가서 12%' + 맛집 링크). 내가 픽한 버튼만 옐로우 solid. 분포 바 없음 — 메뉴판 위계 보호 (2026-06-05 리뷰 반영)
- **나가서 먹기**: 별도 행 없음 — 결과 텍스트에 '맛집 ›' 링크 인라인
- **갭 카드**: 다음날 오전 히어로 아래 1줄 카드 — 라벨(어제의 픽) + 결과 문장 + 블로그 링크
- **정답 판정 기준**: 픽 1위 선택지(테이크아웃·나가서 제외, 코스만)와 만족률 1위 코스가 일치하면 `hit`("정답이었어요"), 불일치면 `miss`("반전 — B가 더 맛있었대요")
- 모바일: 동일 구조 1컬럼, 버튼 2×2 그리드

## 데이터 (Supabase)

```
picks 테이블
- date: date
- choice: text  ('courseA' | 'courseB' | 'takeout' | 'out')
- count: int
- PK (date, choice)

RPC: increment_pick(p_date, p_choice)  — votes의 increment_vote와 동일 패턴
RLS: SELECT anon 허용 / 쓰기는 RPC만
```
- 픽 변경: 클라이언트가 이전 choice를 기억(localStorage) → decrement_pick + increment_pick 또는 change_pick(p_date, p_from, p_to) RPC 1개로 처리. **change_pick 단일 RPC 권장** (원자성)
- 기존 `votes` 테이블·RPC 변경 없음

## 엣지케이스

| 케이스 | 처리 |
|---|---|
| 픽 0명 | 버튼만 (분포는 어차피 투표 후 공개라 별도 처리 불필요) |
| 주말/메뉴 없는 날 | 픽 위젯 미노출 (기존 CLOSED 상태 유지) |
| 오늘이 아닌 날짜 선택(주간 네비) | 픽 위젯·평가 미노출 — 픽/평가는 '오늘' 뷰 한정. 과거 요일은 해당일 평가 결과만 읽기 전용 표시 |
| 13:00 이후 첫 방문 | 픽 기회 없음(행 미노출) — 맛평가만 가능 |
| 마감 직전 제출 (시계 오차) | RPC 서버 시각 검증으로 거부 → "마감되었어요" 토스트, UI 갱신 |
| 자정 직전 평가 제출 | 동일 — 서버 시각 기준 23:59:59까지 유효 |
| 테이크아웃만 있는 날 | 해당 없는 코스 버튼 미노출 (동적 구성) |
| Supabase 실패 | 픽 위젯 비활성 + 기존 투표 실패 패턴 재사용 (메뉴는 정상) |
| 어제 픽 데이터 없음 | 갭 카드 미노출 |
| 평가 0건 | 갭 카드 미노출 (만족률 계산 불가), 자동 블로그는 픽 분포만 기재 |

## Out of Scope (Phase 2 후보)
